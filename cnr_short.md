# Шпаргалка: TypeScript, React, HTTP, REST API

## TypeScript

### 1. TS vs JS
- **JS** = динамическая типизация (в рантайме)
- **TS** = статическая (на этапе компиляции)
- Плюсы TS: раннее обнаружение ошибок, автодополнение, самодокументация, безопасный рефакторинг

### 2. Статическая vs динамическая типизация
- **Динамическая**: типы в рантайме, переменная может менять тип
- **Статическая**: типы при компиляции, тип фиксирован
- **Строгая**: нет неявных преобразований (`"hello" + 5` = ошибка в Python, `"hello5"` в JS)

### 3. Зачем `any`?
- "Аварийный люк" — отключает проверку типов
- Для: миграции с JS, данных без типов, legacy-библиотек
- Избегать где возможно

### 4. `any` vs `unknown`
- `any` — всё разрешено, никаких проверок
- `unknown` — требует проверки типа перед использованием
- `unknown` = type-safe версия `any`

## React

### 5. Почему React популярен?
- Компонентный подход (переиспользование)
- Декларативность (описываем "что", не "как")
- Virtual DOM (эффективные обновления)
- Односторонний поток данных (предсказуемость)
- Огромная экосистема + поддержка Meta

## HTTP/HTTPS

### 6. HTTP vs HTTPS
- **HTTP**: открытый текст, порт 80
- **HTTPS**: шифрование TLS, порт 443
- HTTPS даёт: конфиденциальность, целостность, аутентификацию

### 7. SSL/TLS сертификаты и CA
- Сертификат = подтверждение идентичности + публичный ключ
- **CA** = доверенная организация, подписывает сертификаты
- Цепочка доверия: Root CA → Intermediate CA → Сертификат сайта
- Типы: DV (домен), OV (организация), EV (расширенная проверка)

### 8. Self-signed vs CA-signed
- **Self-signed**: бесплатно, но браузер не доверяет
- **CA-signed**: доверие браузеров из коробки
- Self-signed только для localhost/dev/внутренних сетей

### 9. Риски self-signed
- Уязвимость к MITM (нет подтверждения идентичности)
- Пользователи привыкают игнорировать предупреждения
- Шифрование есть, аутентификации нет

### 10. MITM атака
- Злоумышленник между клиентом и сервером
- Способы: Evil Twin WiFi, ARP Spoofing, DNS Spoofing, SSL Stripping
- HTTPS защищает: шифрование + проверка сертификата

## REST API

### 11. Что такое REST?
- Архитектурный стиль: stateless, клиент-сервер, единообразный интерфейс
- Методы: GET (получить), POST (создать), PUT (заменить), PATCH (обновить), DELETE (удалить)
- Идемпотентность: GET, PUT, DELETE — да; POST — нет

### 12. Почему REST популярен?
- Простота (стандартный HTTP)
- Универсальность (любой HTTP-клиент)
- Кэширование из коробки
- Альтернативы: GraphQL (гибкие запросы), gRPC (производительность), SOAP (enterprise)

### 13. REST для EC2 (пример)
```
GET /instances          — список
GET /instances/i-123    — один инстанс
POST /instances         — создать
PATCH /instances/i-123  — обновить
DELETE /instances/i-123 — удалить
POST /instances/i-123/actions {action: "start"} — действия
```

### 14. API Versioning
- Зачем: обратная совместимость при изменениях
- Способы: URL path (`/v1/users`), query param, header
- URL path — самый распространённый

### 15. Изменения без версионирования
- Аддитивные изменения (добавлять поля, не удалять)
- Deprecation headers
- Feature flags (`?format=extended`)
- HATEOAS (клиенты следуют ссылкам)
